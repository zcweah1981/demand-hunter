# 线索模型库模块方案

> 日期：2026-06-12  
> 状态：讨论确认版  
> 范围：机会发现中的「线索模型库」与后续「线索池」改造方向  

## 1. 命名调整

当前的「来源表现」不再作为菜单名使用。

新的业务命名：

```text
来源表现 → 线索模型库
入口池 → 线索池
入口 → 线索
```

原因：

- 「来源表现」偏内部统计，用户不容易理解；
- 「入口池」表达不够准确，实际承载的是机会线索；
- 系统的第一个口应该是「这些机会线索从哪些模型来」。

## 2. 模块定位

「线索模型库」是机会发现模块的第一个口。

它回答：

```text
系统有哪些模型在发现机会线索？
每个模型从哪里找？
每个模型最近运行了多少次？
每个模型贡献了多少线索？
哪些线索进入候选关键词？
哪些进入关键词库？
哪些生成机会？
哪些模型噪音高或异常多？
哪些模型还未接入？
```

它不是单纯 ROI 报表，也不是证据系统的模型说明页。

它是：

```text
模型视角的机会线索来源管理页
```

## 3. 数据视角

页面要以用户能理解的方式组织数据：

```text
来源类别
  ↓
线索模型
  ↓
模型明细
  ↓
每次自动化 / 人工运行记录
  ↓
本次产生的线索和后续效果
```

每个模型都要能看：

```text
模型状态
模型作用
输入是什么
输出是什么
最近运行时间
搜索 / 抓取次数
抓取结果数
贡献线索数
候选关键词数
进入关键词库数
生成机会数
Action / Watch / Reject 数
异常数
噪音率
```

未接入的模型也要展示，状态写：

```text
未接入
```

这样用户知道系统规划里有这个模型，只是当前还没有启用。

## 4. 模型分类

第一版按以下类别展示：

```text
搜索扩展类
SERP 类
站点类
热点 / 早期信号类
社交 / 论坛 / 评论类
文档 / 更新 / 定价类
四找闭环类
```

### 4.1 搜索扩展类

```text
Google Suggest
Short Tail Rewrite
Root Combo
词找词
```

作用：

```text
从已有搜索词、短词、联想词中扩展更多成熟搜索需求。
```

### 4.2 SERP 类

```text
SERP Search
Advanced Search
词找站
```

作用：

```text
从搜索结果中发现承接需求的网站、竞品、页面和内容缺口。
```

### 4.3 站点类

```text
Sitemap
Domain Web
Alternative
站找词
站找站
```

作用：

```text
从站点、竞品、替代品、对比页、新页面中发现机会线索。
```

### 4.4 热点 / 早期信号类

```text
Hot Topic
HN Algolia
GitHub Trending
Product Hunt
Steam
arXiv
```

作用：

```text
捕捉刚出现的新工具、新项目、新游戏、新模型、新平台变化。
```

### 4.5 社交 / 论坛 / 评论类

```text
Social
Forum
Review
Reddit
Community Watch
```

作用：

```text
从用户讨论、抱怨、评价、论坛帖子中发现真实需求和痛点。
```

### 4.6 文档 / 更新 / 定价类

```text
Docs
Changelog
Pricing Pages
```

作用：

```text
从产品更新、文档变化、定价变化中发现新机会、商业模式变化和竞品动作。
```

### 4.7 四找闭环类

```text
词找词
词找站
站找词
站找站
```

四找可以同时属于具体类别，但也需要作为完整闭环被看见。

它要能回答：

```text
从哪个 seed 出发？
扩展了哪些词？
发现了哪些站？
从站点反查了哪些词？
又发现了哪些相邻站？
最终形成了哪些线索和机会？
```

## 5. 人工与自动化

如果某个模型支持人工添加来源，就把人工入口放在该模型模块里。

例如 Sitemap：

```text
Sitemap 模型
  - 人工添加域名
  - 人工添加 sitemap URL
  - 立即运行
  - 下一轮自动化运行
```

人工与自动化进入同一条链路，但数据必须区分来源：

```text
输入来源：manual / automation / system
运行触发：manual_run / scheduled_run / automation_cycle
```

后续统计时可以分别看：

```text
人工贡献了多少线索
自动化贡献了多少线索
人工输入后自动化产生了哪些后续结果
```

## 6. 模型明细交互

模型明细使用抽屉弹窗，不单独跳转页面。

点击模型卡片后，从右侧打开抽屉。

抽屉内容包括：

```text
模型说明
模型状态
人工输入入口
最近运行记录
每次运行输入
每次运行结果
产生的线索
候选关键词转化
关键词库转化
机会卡转化
Action / Watch / Reject
异常和过滤原因
```

抽屉中的运行记录需要有时间维度。

示例：

```text
Sitemap

运行 #120
时间：2026-06-11 02:00
触发：automation_cycle
输入：expiryedge.com, taxjar.com
抓取 URL：80
新页面：12
保存线索：4
候选关键词：3
进入关键词库：1
拒绝：8
异常：0
```

## 7. 原「来源 × 搜索条件效果矩阵」调整

原来的：

```text
来源 × 搜索条件效果矩阵
```

不再作为页面底部独立模块。

它的实际含义是：

```text
某个模型围绕某个输入对象，在某一次或多次运行中产生了什么效果。
```

所以它应该进入模型明细抽屉，改名为：

```text
运行效果明细
```

或：

```text
输入对象效果
```

字段需要更用户化：

```text
时间
运行批次
输入对象
模型
抓取结果
保存线索
有效线索
拒绝线索
候选关键词
关键词库
机会
异常
判断
```

这样用户可以看懂：

```text
这一次自动化到底做了什么？
它为什么说这个模型有效？
哪些结果是人工来的？
哪些结果是自动来的？
哪些被过滤？
为什么被过滤？
```

## 8. 与线索池的关系

「线索模型库」回答：

```text
线索从哪里来？
模型如何产生线索？
每个模型贡献和效果如何？
```

「线索池」回答：

```text
已经产生了哪些线索？
每条线索是什么类型？
来自哪个模型？
关联哪条证据？
当前状态是什么？
下一步应该评分、补证、转译还是排除？
```

两者关系：

```text
线索模型库
  ↓ 产生
线索池
  ↓ 转译 / 评分 / 补证
候选关键词
  ↓ 质量门
关键词库
  ↓ SERP / 竞品 / MVP / 变现分析
机会卡
```

## 9. 第一版实现方向

第一版先实现：

```text
1. 菜单改名：来源表现 → 线索模型库，入口池 → 线索池
2. 线索模型库按类别展示模型
3. 已有模型显示真实统计
4. 未接入模型显示未接入状态
5. 模型点击打开右侧抽屉
6. 抽屉展示模型说明、状态、运行记录、运行效果明细
7. Sitemap 等已有可运行模型提供人工输入入口
8. 原矩阵数据迁入模型抽屉，不再放页面底部
```

暂不在第一版解决：

```text
线索池完整重构
所有未接入模型真实抓取
复杂来源权重自动调整
完整人工 / 自动化贡献归因
```

这些在「线索模型库」第一版确认后，再继续讨论和实现。
