# 线索池重构方案与实施计划

> 日期：2026-06-12  
> 状态：讨论确认版  
> 范围：机会发现中的「线索池」模块；自动化全流程暂不纳入本阶段实现  

## 1. 背景与结论

当前系统已经有「线索模型库」的方向：模型负责从搜索扩展、SERP、站点、热点、社交、文档更新和四找闭环中发现机会线索。

但「线索池」目前还没有形成清晰的用户视图，用户看到的是候选关键词或内部状态，容易变成黑箱。我们需要把线索池重构为：

```text
所有模型产出的机会线索的统一工作台
```

线索池不负责解释每个模型怎么运行，那是线索模型库的职责。线索池负责回答：

```text
系统已经发现了哪些线索？
线索来自哪个模型？
线索来自哪个输入对象？
线索为什么有价值？
线索当前处在什么状态？
线索下一步会去哪里？
线索是否已经进入关键词库或机会分析？
```

## 2. 模块边界

### 2.1 线索模型库

线索模型库回答「线索从哪里来」。

它关注：

```text
模型分类
模型状态
模型说明
输入对象
运行记录
产生线索
异常
模型贡献
```

### 2.2 线索池

线索池回答「线索本身怎么样」。

它关注：

```text
线索对象
线索类型
来源模型
输入对象
需求分
趋势分
总评分
质量门
当前状态
去向
追溯明细
```

### 2.3 候选关键词

候选关键词不应继续作为用户难以理解的黑箱模块。

新的定位是：

```text
线索池中的一个状态
```

即：

```text
某条线索词通过基础质量门后，进入「候选关键词」状态，准备进入正式关键词库。
```

候选关键词仍然可以在数据层存在，但用户视角应从线索池理解它，而不是把它当成独立流程。

## 3. 核心对象关系

线索池的核心关系是：

```text
输入对象
  ↓ 被模型使用
线索模型
  ↓ 产生
线索
  ↓ 清洗 / 评分 / 门禁
候选关键词状态
  ↓ 入库
关键词库
  ↓ SERP / 竞品 / 证据 / 机会分析
机会
```

一条线索必须尽量能追溯到：

```text
哪个模型产生
哪一次运行产生
哪个输入对象产生
产生时的上下文
后续经过哪些评分和门禁
最终去了哪里
```

历史数据如果缺少 `inputRef`，页面可以明确显示「历史记录未保存输入对象」，但后续新运行必须补齐。

## 4. 线索类型

第一阶段线索池以「线索词」为主，但数据结构和页面语言要为更多线索类型留出空间。

线索类型包括：

```text
搜索词
网站
页面
趋势实体
社区话题
产品变化
定价变化
更新记录
```

当前可以优先落地：

```text
搜索词
网站
页面
```

趋势实体、社区话题、产品变化、定价变化、更新记录可以先作为规划类型展示，等对应模型接入后再真实产生。

## 5. 状态设计

线索状态应该表达系统推进位置，而不是暴露内部字段。

建议状态：

```text
新发现
待评分
待补证据
候选关键词
已入关键词库
已生成机会
已过滤
需人工处理
```

### 5.1 新发现

模型刚产生，已经进入线索池，但还没有完成清洗、去重和评分。

### 5.2 待评分

线索已经通过基础清洗，等待计算需求分、趋势分和质量门结果。

### 5.3 待补证据

线索有潜力，但证据不足，无法直接进入关键词库或机会分析。

### 5.4 候选关键词

线索词通过基础质量门，准备进入正式关键词库。

这是候选关键词的新定位：它不是独立黑箱，而是线索池状态。

### 5.5 已入关键词库

线索已经进入正式关键词库，后续可以参与 SERP、竞品、机会评分。

### 5.6 已生成机会

线索已经生成机会对象或关联已有机会。

### 5.7 已过滤

线索被系统或人工排除，原因包括噪音、重复、品牌残留、太泛、意图不清等。

### 5.8 需人工处理

线索出现冲突或异常，例如：

```text
评分冲突
重复关系不清
来源异常
模型输出异常
证据不足但分数较高
```

## 6. 状态推进方式

线索状态不应该依赖用户逐条手动推进。

当前我们确认：

```text
系统未来应以一个大周期自动运行，状态在每一轮运行中批量推进。
```

但自动化全流程暂时放到后面重构。本阶段线索池先按这个目标设计页面和数据，不强行完成全自动执行。

第一阶段的状态推进原则：

```text
系统自动判断为主
人工只处理异常、灰区和纠偏
```

人工按钮根据页面语境出现，不做大量通用按钮。

示例：

```text
高分未入库：推进
证据不足：补证据
分数异常：重新计算
明显错误：排除
已过滤：恢复
异常记录：处理异常
```

## 7. 评分设计

线索池需要把「需求质量」和「趋势质量」分开，不再只给一个含糊总分。

### 7.1 需求分

需求分回答：

```text
这是不是一个明确、可承接、可能商业化的需求？
```

参考因素：

```text
任务意图是否明确
是否有工具 / 模板 / 计算器 / 自动化等意图
是否过泛或过短
是否有商业意图
是否符合当前系统关注领域
是否被质量门判定为噪音
```

### 7.2 趋势分

趋势分回答：

```text
这个线索是不是正在变新、变热、变重要？
```

参考因素：

```text
是否来自新页面 / 新 sitemap
是否来自 changelog / docs / pricing 变化
是否来自热点平台或社区讨论
是否多轮重复出现
是否跨模型重复出现
是否在近期频繁增长
```

### 7.3 总评分

总评分不应该替代需求分和趋势分，而是用于排序和状态推进。

页面要能解释：

```text
需求分为什么高或低
趋势分为什么高或低
质量门是否通过
扣分原因是什么
当前建议是什么
```

## 8. 表格视图

线索池主表必须按用户视角设计，默认不展示原始 JSON。

主表建议字段：

```text
线索
类型
状态
来源模型
输入对象
需求分
趋势分
总评分
质量门
关键词库状态
机会状态
最近发现
下一步
```

表格必须支持：

```text
分类
过滤
筛选
排序
追溯
```

常用筛选：

```text
按状态筛选
按类型筛选
按来源模型筛选
按输入对象筛选
按分数区间筛选
按质量门结果筛选
按是否入关键词库筛选
按是否生成机会筛选
按异常筛选
```

常用排序：

```text
总评分
需求分
趋势分
最近发现时间
模型贡献次数
证据数量
异常数量
```

## 9. 明细抽屉

点击线索后打开明细抽屉。

明细抽屉应该回答：

```text
这条线索是什么？
为什么出现？
谁发现的？
为什么当前是这个状态？
为什么是这个分数？
后续去了哪里？
需要人工做什么？
```

建议分区：

```text
线索概览
来源追溯
评分说明
质量门
状态时间线
关联关键词
关联机会
证据与变化
异常与处理
```

### 9.1 来源追溯

展示：

```text
来源模型
运行批次
输入对象
触发方式
产生时间
同批次产生的其他线索
```

### 9.2 评分说明

展示：

```text
需求分构成
趋势分构成
加分项
扣分项
质量门结果
当前建议
```

### 9.3 状态时间线

展示每次关键变化：

```text
新发现
完成评分
补充证据
进入候选关键词
进入关键词库
生成机会
被过滤
人工恢复
```

每个节点都要写清：

```text
发生时间
触发来源
变化前状态
变化后状态
依据
相关证据
```

## 10. 与线索模型库的衔接

线索模型库中的「运行记录」和线索池中的「来源追溯」必须能互相对应。

对应关系：

```text
线索模型库：从模型和运行批次看结果
线索池：从单条线索看来源和后续去向
```

同一条线索在两个页面看到的事实必须一致：

```text
modelKey
runId
inputRef
generatedClue
status
score
target keyword
target opportunity
```

## 11. 与关键词库的衔接

关键词库不应该接收来源不清的词。

进入关键词库前，线索至少应该具备：

```text
来源模型
输入对象或明确历史缺失说明
需求分
趋势分
质量门结果
状态变化记录
```

进入关键词库后，线索池显示：

```text
已入关键词库
关键词 ID
入库时间
后续 SERP / 机会状态
```

## 12. 与证据系统的衔接

证据仍然保持客观，不直接等同于评分。

线索池使用证据的方式是：

```text
展示这条线索有哪些客观证据
说明哪些证据被用于需求分或趋势分参考
说明证据服务于哪个对象
```

证据可以服务于：

```text
输入对象
线索
关键词
机会
监控对象
```

但评分解释必须明确：

```text
证据是事实
评分是系统基于事实做出的判断
```

## 13. 数据要求

线索池重构需要逐步统一以下记录。

### 13.1 线索记录

每条线索建议具备：

```text
id
clueType
value
normalizedValue
status
demandScore
trendScore
totalScore
qualityGate
sourceModel
sourceRunId
inputRef
generatedAt
lastSeenAt
keywordId
opportunityId
evidenceRefs
timeline
```

### 13.2 来源追溯

每条新线索必须记录：

```text
modelKey
runId
inputRef
trigger
generatedClue
```

### 13.3 异常记录

异常不混在线索明细默认表里，应单独查看和处理。

异常记录至少包含：

```text
runId
modelKey
inputRef
message
retryable
occurredAt
status
resolution
```

## 14. 第一阶段实施计划

第一阶段只做线索池，不做完整自动化重构。

### 阶段 1：梳理现有数据映射

目标：

```text
明确当前 candidate_keywords、keywords、SourceRun、generatedClues、inputRef 如何映射到线索池。
```

验证：

```text
能列出线索池主表每一列来自哪个现有字段或需要新增哪个字段。
```

### 阶段 2：定义线索池 API 输出

目标：

```text
新增或改造一个面向前端的线索池 API，输出用户视角字段，而不是直接返回内部表。
```

验证：

```text
测试覆盖状态映射、来源追溯、评分字段、历史 inputRef 缺失说明。
```

### 阶段 3：主表页面重构

目标：

```text
把线索池主页面改成可分类、过滤、筛选、排序、追溯的表格。
```

验证：

```text
用户能从主表看到线索、状态、来源模型、输入对象、需求分、趋势分、关键词库状态和机会状态。
```

### 阶段 4：线索明细抽屉

目标：

```text
点击线索打开明细抽屉，展示来源追溯、评分说明、质量门、状态时间线和关联对象。
```

验证：

```text
一条线索能追溯到模型运行记录；历史缺失数据明确显示原因，不猜测。
```

### 阶段 5：候选关键词状态合并

目标：

```text
把候选关键词作为线索池状态表达，弱化独立黑箱页面。
```

验证：

```text
用户能理解某条线索为什么是候选关键词，以及它进入关键词库还缺什么。
```

### 阶段 6：评分说明与质量门

目标：

```text
在线索明细中解释需求分、趋势分、总评分和质量门。
```

验证：

```text
每条线索至少展示分数来源、加分项、扣分项、质量门结果和建议动作。
```

### 阶段 7：与模型库记录互通

目标：

```text
线索池明细能回到对应模型、运行批次和输入对象；模型库运行记录能看到产生的线索。
```

验证：

```text
同一条线索在模型库和线索池中的来源信息一致。
```

### 阶段 8：异常与人工处理入口

目标：

```text
异常独立展示，人工只处理异常、灰区和纠偏，不承担常规推进。
```

验证：

```text
异常有独立查看入口；需要处理时出现语境按钮，例如处理异常、重新计算、补证据、排除、恢复。
```

## 15. 暂不纳入本阶段

以下内容后续在自动化全流程重构时处理：

```text
统一大定时器内完整执行所有模块
所有模型真实接入
所有采集器强制写入 generatedClues / inputRef / errors
证据系统全链路回流
机会自动重评分
专用周期拆分
```

本阶段只要求线索池的用户视角、数据映射、追溯结构和页面计划清楚。

## 16. 成功标准

线索池重构完成后，用户应该能回答：

```text
现在系统发现了哪些线索？
哪条线索最值得继续？
它来自哪个模型？
它来自哪个输入对象？
为什么需求分高？
为什么趋势分高？
为什么被过滤或进入候选关键词？
它有没有进入关键词库？
它有没有生成机会？
如果出问题，应该处理什么异常？
```

如果这些问题需要看原始 JSON 或猜测内部状态，说明线索池还没有重构成功。
